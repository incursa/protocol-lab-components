# Code Of Conduct

Incursa projects are open-source workspaces for practical software
collaboration. Keep discussion technical, specific, and respectful.

## Expected Behavior

Participants are expected to:

1. Be direct without being abusive.
2. Assume good intent until behavior proves otherwise.
3. Critique ideas, component behavior, package metadata, docs, and tradeoffs
   rather than attacking people.
4. Keep issues and pull requests focused on this repository's component package
   scope.
5. Respect privacy. Do not post private transcripts, credentials, customer
   data, private paths, or personal contact details.
6. Disclose conflicts of interest when they materially affect a contribution
   or recommendation.

## Unacceptable Behavior

The following behavior is not acceptable:

1. Harassment, threats, stalking, or intimidation.
2. Personal attacks, slurs, or discriminatory comments.
3. Sexualized language or imagery in project spaces.
4. Posting another person's private information without permission.
5. Deliberately derailing issues, pull requests, or reviews.
6. Submitting malware, credential material, exploit payloads, or intentionally
   deceptive content.
7. Retaliation against someone who reports a concern in good faith.

## Scope

This code applies in repository spaces, project communication channels, issue
and pull request discussions, and related public interactions where someone is
representing the project.

## Reporting

Report conduct concerns to oss@incursa.com.

For security vulnerabilities, follow [SECURITY.md](SECURITY.md) instead of
posting details publicly.

Reports should include:

1. A short description of the behavior.
2. Links, screenshots, or timestamps when available.
3. Whether there is an immediate safety, privacy, or security concern.

Reports will be reviewed by the project owner. The response may include
clarification, moderation, warning, temporary restrictions, permanent removal
from project spaces, or referral to the relevant platform.

## Enforcement

Maintainers may remove comments, close issues, restrict participation, or block
accounts when necessary to keep the project usable and safe.

Enforcement decisions should be proportional to the behavior, the impact on
contributors, and the risk to users.
